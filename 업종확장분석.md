# Skima Byte 업종 확장 분석

작성일: 2026-05-03
대상: 카페 외 업종 적용 가능성 — 특히 유치원·돌봄·방과후 영역

---

## 1. 유치원·돌봄·방과후 영역 — 적합도 평가

### 1.1 적합한 이유 (가치 제안이 카페보다 더 강력)

| 항목 | 카페 | 어린이집·돌봄 |
|---|---|---|
| 시급 | 1만~1.3만 | **2.5만~4만** (대체교사) |
| 펑크 임팩트 | 매장 운영 차질 | **인가 정원 위반·부모 항의** (법적 사고) |
| "1시간 매칭" 절박도 | 중 | **상** (당일 아침 펑크 → 9시 등원 전 확보 필수) |
| 거래당 수수료 매출 | 낮음 | **3~4배 높음** |
| 양방향 평가 가치 | 보통 | **매우 큼** (신뢰가 곧 안전) |

→ "노쇼 0%" + "1시간 매칭"이라는 정체성이 **카페보다 훨씬 절박하게 팔리는 시장**.

### 1.2 진입 비용 (카페와 차원이 다름)

1. **자격증 자기신고 불가**
   - 현재 `User.certifications`는 `Set<String>` 자기신고
   - 보육교사·유치원정교사는 **자격증 사진 업로드 + 보건복지부 자격관리시스템 검증** 또는 최소한 점주 수동 검증 필수
2. **성범죄·아동학대 경력 조회 의무** (아동복지법 §29의3)
   - 채용 전 신원조회 의무 — 플랫폼이 이 흐름을 끼지 않으면 점주가 안 씀
3. **사고 발생 시 책임 소재**
   - 영유아 사고는 플랫폼 책임론으로 번지기 쉬움
   - PL 보험 또는 약관상 책임 분리 필수
4. **B2B 영업 사이클이 길다**
   - 원장은 보수적
   - "긴급 대체교사 풀 앱" 같은 명확한 wedge로 시작 (정규 채용 대체 아님)

### 1.3 도메인 모델 변경 비용 (생각보다 가벼움)

현재 구조의 **80%가 그대로 재사용 가능**:

**그대로 쓰는 것**
- `Shift` (시간/시급/headcount/status)
- `ShiftMatch`, `Payout`, `Rating`
- GPS 체크인, 채팅, 노쇼 백업, 알림

**일반화 필요한 것**
- `Cafe` → `Workplace` (또는 `Cafe` 유지 + `WorkplaceType` 추가)
- `CafeType` enum → 업종 카테고리 확장
- `JobRole` enum이 카페 직무로 하드코딩 (BARISTA/HALL/...) → 업종별 직무 트리 필요
- `Requirements`는 이미 `Set<String>` 자유형이라 확장 가능

**신규 필요**
- `CredentialVerification` (자격증 검증)
- `BackgroundCheck` (신원조회)

→ 새 프로젝트가 아니라 **"Skima Care" 같은 vertical 한 줄 추가**로 갈 수 있음.

## 2. Skima Byte 모델에 적합한 업종 매트릭스

평가 기준: **(긴급성) × (시급단가) × (규제 부담↓) × (1회성 적합도) × (현 도메인 재사용도)**

| 업종 | 적합도 | 핵심 이유 |
|---|---|---|
| **베이커리·디저트샵** | ★★★★★ | 이미 카탈로그에 있음. 거의 코드 변경 0. 현재 MVP가 사실상 커버 |
| **음식점 홀/주방** | ★★★★★ | JobRole만 추가(SERVER/DISHWASHER/PREP). 시장 규모는 카페의 5~10배 |
| **편의점·무인매장** | ★★★★☆ | 야간 펑크 빈발, GPS+CCTV로 부정 적음. 단가는 낮음 |
| **펍·와인바·이자카야** | ★★★★☆ | 저녁 피크 스팟 수요 강함. 시급도 카페보다 ↑ |
| **행사·팝업·전시 스태프** | ★★★★☆ | 본질이 1회성. "주말 페어 안내요원 50명" 같은 bulk 매칭 |
| **돌봄·방과후 (보조)** | ★★★★☆ | 위 분석. 검증 인프라 추가 필요하지만 단가·임팩트 최고 |
| **미용실 샴푸·보조** | ★★★☆☆ | 시간대 명확. "단골 미용사" 의존도가 커서 1회성과 결이 어긋남 |
| **물류·택배 단기 분류** | ★★★☆☆ | 시장 거대하지만 쿠팡플렉스/배민커넥트 dominant |
| **청소·하우스키핑** | ★★★☆☆ | 청소연구소·미소가 이미 점유. 차별화 필요 |
| **학원 보조·시험감독** | ★★★☆☆ | 시급 높지만 빈도 낮음. 정기시즌 기반 |
| **의료·약국** | ★★☆☆☆ | 면허 검증 부담 너무 큼 |
| **배달 라이더** | ★☆☆☆☆ | 레드오션 |

## 3. 진행 방향 옵션

### A. 음식점·펍으로 수평 확장 (저위험·저보상)
- `JobRole` enum 확장 + `CafeType`에 RESTAURANT/PUB 추가
- 작업량: **1~2일**
- 도메인 재사용: 99%
- PMF 신호: 카페와 비슷할 듯

### B. 어린이집 대체교사 vertical (고위험·고보상)
- "Skima Care" 분기
- 자격증 업로드·검증 흐름 + 약관 + 보험 검토 필요
- 작업량: **2~3주**
- 단가·임팩트는 최고지만 첫 점주 영업이 어려움

### C. 멀티 vertical 플랫폼화 (대공사)
- `WorkplaceType` 추상화 + 업종별 직무 트리·요구자격 메타데이터
- 작업량: **4~6주**
- 향후 어떤 업종이든 카탈로그만 추가하면 운영 가능
- MVP 검증 끝나기 전 너무 이른 abstraction일 수도

---

## 4. 추천 순서

**A 먼저 (1~2일) → B를 별도 vertical로 검증 (2~3주)** 순서 권장. C는 B에서 진짜 갭이 발견된 후에.

**이유**
- A는 거의 공짜로 시장 5배 확장 → MVP의 PMF 신호를 더 빨리 모음
- B는 단가·임팩트가 압도적이지만 검증·보험·영업이라는 비-코딩 작업이 큼 → 별도 트랙으로 다뤄야 일정이 꼬이지 않음
- C는 B를 직접 해보고 나서 진짜 공통 추상화가 뭔지 보고 결정 (premature abstraction 위험)

## 5. 결정 후 첫 작업 메모

### A 선택 시
- `CafeType` enum: RESTAURANT / PUB / IZAKAYA / WINE_BAR 추가
- `JobRole` enum: SERVER / DISHWASHER / PREP_COOK / BARTENDER 추가
- 브랜드 카탈로그에 음식점·펍 프리셋 추가
- 시드 데이터에 음식점 1~2곳 + 시프트 샘플
- 워커 능력 자기신고 UI에서 직무 칩 늘어남에 따른 레이아웃 검토

### B 선택 시
- 신규 도메인: `Workplace` (또는 `CareCenter` 별도) + `WorkplaceType`
- 신규 도메인: `CredentialVerification`, `BackgroundCheckReport`
- 워커 온보딩 흐름 재설계 (자격증 업로드 → 점주 검토 → 활성화)
- 점주 측 검증 UI (자격증·신원조회 보고서 열람·승인)
- 약관·개인정보 처리 동의 별도 (민감정보 다수)
- 보험 상품 리서치 (PL 보험, 단기 근로자 상해)
- 첫 파일럿 어린이집 1~2곳 영업 (제품 만들기 전 인터뷰 우선)

## 6. Skima Care 세부 영역 분석 (보육 / 유치원 / 방과후)

B vertical을 들어간다고 가정하고 세 영역 비교.

| 차원 | **보육 (영유아)** | **유치원 대체** | **방과후 (초등)** |
|---|---|---|---|
| 대상 | 0~5세 어린이집 | 5~7세 유치원 | 6~13세 초등 |
| **펑크 절박도** | ★★★★★ 인가 정원 위반 = 운영정지 | ★★★★ 정원 위반 + 학부모 즉시 항의 | ★★ 1회 결강은 보강 가능 |
| 자격 요건 | 보육교사 2급 필수 | 유치원정교사 (더 엄격) | 특기 분야 자격 (보육자격 불요 多) |
| 신원조회 | 필수 | 필수 | 필수 |
| 시급 | 2.5~3.5만 | 3~4만 | 특기 2.5~4만 / 돌봄 1.5~2만 |
| **시장 구조** | 분산 (전국 3.5만 어린이집) | 분산 (사립 ~4천) | **B2B2C** (위탁업체 BFC·한솔·늘봄이 중간) |
| **영업 난이도** | 한 곳씩 뚫어야 함 | 한 곳씩 + 의사결정 느림 | 위탁업체 1곳 = 학교 100개 |
| 사고 리스크 | **★★★★★ 회사 망함** | ★★★★ | ★★ |

### 영역별 wedge 가설
- **보육·유치원**: "9시 등원 전 1시간 대체" — Skima Byte 정체성 그대로 살아남는 진짜 wedge
- **방과후**: 펑크 절박도 낮음 → wedge가 "긴급 대체"가 아니라 **"검증 강사 풀 + B2B2C 매칭"**으로 변형됨. Skima Byte 시간 SLA·노쇼 자동백업이 절반은 안 쓰임

### 매트릭스
|  | 긴급 대체 | 강사 풀 |
|---|---|---|
| 보육·유치원 | **★★★★★ 진짜 wedge** | ★★ (정규 채용 — 우리 영역 아님) |
| 방과후 | ★★ (절박 낮음) | ★★★★ (B2B2C 스케일) |

→ Skima Care 진입한다면 **보육·유치원 "긴급 대체"가 진짜 wedge**. 방과후는 Phase 2 확장.

### 결정의 갈등
- 사용자 보유 채널: 방과후 교사만 있음 (보육·유치원 0)
- 진짜 wedge는 보육·유치원
- → 방과후 인터뷰를 정찰로 쓰되 "보육·유치원 라인 소개 가능?" 질문 무조건 추가하는 게 합리적
