## 2. 추천 근거

### Segment: 왜 펍·이자카야·와인바인가
1. **wedge 강도 카페보다 강함** — 저녁 피크 펑크는 한낮 카페 펑크와 차원이 다름. "노쇼 0%"가 nice-to-have → must-have로 격상
2. **시급 1.3~1.5배** (1.5~1.8만) → 거래당 수수료 매출 비례 ↑
3. **카페 모델 일반화 가설을 가장 빨리 검증** — 같은 메커니즘이 다른 시간대·업종에서 작동하는지 1~2주면 데이터로 보임
4. **점주 의사결정 빠름** (개인 사장 위주) → 영업 사이클이 카페와 동일

### Segment: 왜 음식점 통째로(b)는 비추천인가
- wedge 흐려져 "또 하나의 알바 플랫폼"이 될 위험
- 분식·김밥집은 카페와 너무 비슷해서 새로운 학습 없음
- 프랜차이즈는 본사 결정 — 영업 사이클 길어짐

### 카피: 왜 카페 메인 유지(c)인가
- 기존 카페 점주·워커가 혼란스러우면 안 됨 (현재 시드·UI 모두 카페 중심)
- 펍 segment가 데이터로 검증되기 전 리브랜딩은 매몰비용
- 검증 후 비중이 역전되면 그때 리브랜딩 (예: "Skima — 카페·펍 스팟워크")

## 3. Phase 1 — 코딩 (1~2일)

### 백엔드
- `CafeType` enum: `PUB` / `IZAKAYA` / `WINE_BAR` 추가
  - 음식점(`RESTAURANT`)은 일단 보류
- `JobRole` enum: `SERVER` / `BARTENDER` / `KITCHEN_HELPER` 추가
- 브랜드 카탈로그에 펍·이자카야·와인바 프리셋 10~15종 (letter+color 아바타)
- `DataSeeder` 시드: 펍 1~2곳 + 저녁 시간대 시프트 샘플
- 평가 기본값·컴플라이언스 룰은 카페와 동일 (변경 없음)

### 프론트
- 워커 능력 자기신고 UI 직무 칩 레이아웃 검토 (개수 늘어남에 따른 wrap)
- 매장 등록 모달의 업종 선택 UX 검토 (카페·베이커리·펍 그룹핑)
- 시프트 검색·카드에서 업종별 시각 차별화 검토 (필요 시)
- 카피는 카페 메인 유지, 신규 업종은 자연스럽게 추가만

### 비-코딩
- 작명 검토 — "Skima Byte"가 카페 외 업종까지 자연스럽게 포괄하는지

## 4. Phase 2 — 시장 검증 (1~2주)

### 영업
- 펍·이자카야·와인바 점주 3~5곳 영업
- 실제 시프트 등록 → 워커 매칭 → 정산 e2e 사이클 1바퀴

### 측정 지표
- 시프트 등록 수 / 카페 대비 비율
- 펑크 발생 빈도 (가설: 카페보다 ↑)
- 노쇼 자동백업 발동률
- 점주 재등록률 (1주 후)
- 워커 지원율 (저녁 시간대 공급 확보되는지)

### 결정 분기 (1~2주 후)
- **잘 됨** → 호프집·고깃집까지 확장 → 일반 음식점으로 점진 확대
- **그저 그럼** → 펍 segment 좁히고 다른 segment 시도 (호프·고깃집 직접)
- **안 됨** → 카페로 회귀, 카페 내 깊이 파기 (제품 고도화 로드맵 4축)
